Permission Anchor Lost When Location Bar Hidden

From permission_prompt_bubble_view_browsertest.cc:

// Test switching between chip and bubble — make sure no crashes.
browser_view->GetLocationBarView()->SetVisible(false); // ← triggers anchor fallback
ShowUi("geolocation");
permission_request_manager->UpdateAnchor(); // ← anchor now at page-level
browser_view->GetLocationBarView()->SetVisible(true);
permission_request_manager->UpdateAnchor(); // ← chip visible again

When LocationBarView is hidden (zero height, or visibility toggled), UpdateAnchorPosition() falls back to GetPageInfoAnchorRect(browser_) — which is inside the page's coordinate frame. A permission bubble now floats within the page content area instead of the browser chrome.

🎯 Clickjacking Attack Chain

1
Page requests permission (geolocation/camera). Bubble appears anchored to location bar at top of screen.
2
Page uses CSS animation/scroll to visually push browser chrome out of viewport (e.g. extreme zoom, scroll-to-top-button, or kiosk mode).
3
When LocationBarView becomes invisible/zero-height, UpdateAnchor() is called and the permission bubble RE-ANCHORS to the page's GetPageInfoAnchorRect.
4
The bubble now floats inside the page content. Attacker positions an opaque UI element OVER the "Allow" button area, tricking the user into clicking it.

Testing methodology: Combine with kiosk-mode or PWA standalone windows where the location bar is already hidden by design (Apps use is_type_app() which doesn't have a location bar). Permission prompts in PWA windows may re-anchor differently.
